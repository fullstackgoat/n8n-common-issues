# Repro 08 — HTTP Request: `ECONNREFUSED`, proxies & self-signed certs

Three forum-canonical HTTP failures, one compose stack, three workflow JSONs. Every bug looks like *"n8n's HTTP node is broken"*; every fix is one or two env vars on the n8n container.

## Symptoms (any combination)

- *"n8n can't reach Ollama running on my host machine"* — `ECONNREFUSED 127.0.0.1:11434`
- *"My workflow can't reach our internal API behind the corporate proxy"* — `ETIMEDOUT` or `ENOTFOUND`
- *"HTTP Request fails with `self-signed certificate in certificate chain` / `unable to verify the first certificate`"*
- *"Works on my laptop, fails when I run n8n in Docker"*
- The HTTP node renders the request fine in *Test step*, but the same workflow fails when *Executed*

## What this repro proves

That all three symptoms above stem from the **n8n container's network and trust environment**, not from the HTTP Request node itself. Three orthogonal fixes:

| Scenario | Bug | Fix |
|---|---|---|
| 1 — Ollama-on-host | `localhost` inside the container is the container itself, not the host | Use `host.docker.internal` and add `extra_hosts: ["host.docker.internal:host-gateway"]` (Linux only) |
| 2 — Corporate proxy | n8n has no route to internal services | Set `HTTP_PROXY` and `HTTPS_PROXY` env vars to point at your proxy |
| 3 — Self-signed cert | Node.js rejects untrusted certs by default | Mount the CA bundle and set `NODE_EXTRA_CA_CERTS` (NOT `NODE_TLS_REJECT_UNAUTHORIZED=0`, except in dev) |

## Stack architecture

```
                             +---------------- public network ----------------+
                             |                                                |
   host:11434  <--published--+--> fake-ollama (whoami)                        |
                             |                                                |
                             |   n8n  <--->  squid  <--->  self-signed-nginx  |
                             |                |                               |
                             +----------------|-------------------------------+
                                              v
                             +-------- private network --------+
                             |  squid (also)  <--->  internal-api (whoami)
                             +---------------------------------+

   cert-init (oneshot, alpine/openssl) --writes--> [certs volume]
                                                        |
                                          +-------------+-------------+
                                          v                           v
                                  self-signed-nginx               n8n (fixed only,
                                  (TLS server)                    via NODE_EXTRA_CA_CERTS)
```

Why this design:

- **`fake-ollama` is published to the host on `11434:80`** so that `http://host.docker.internal:11434/` from inside the n8n container actually resolves to a real service. That's what makes the scenario-1 fix verifiable.
- **`internal-api` is on the `private` network only** — n8n on `public` cannot reach it via Docker DNS. With `HTTP_PROXY=http://squid:3128` set, n8n forwards to Squid (which is on both networks) and Squid resolves and fetches `internal-api`. That's the scenario-2 demonstration.
- **`cert-init` generates a self-signed cert into a named volume** on first boot. Both `self-signed-nginx` and (in the fixed stack) the n8n container mount it read-only. No keys live in the repo.

## Quick start

```bash
cd repros/08-http-econnrefused
cp .env.example .env

# 1) Boot the broken stack
docker compose up -d
docker compose logs -f n8n | grep "Editor is now accessible"
# Press Ctrl-C once you see it
```

Open <http://localhost:5678>, create an owner account, then import any of the three workflow JSONs and **Test workflow**. Each one fails in a different, instructive way.

```bash
# 2) Switch to the fixed stack
docker compose down
docker compose -f docker-compose.fixed.yml up -d
```

Re-run the same workflows. All three succeed.
