Build #1
07
Wave 1 · Day 1 Budget ~2h

Loop "first item only"

Repros mental-model bug · Pure workflow JSON, no infra

The problem

A customer wires up a workflow with five input items. Only the first item produces a downstream effect. They wrap the work in a "Loop Over Items" node — nothing changes. Then they hit "Input values have 3 keys" when calling a sub-workflow.

The bug isn't in n8n. The customer's mental model of how items flow is wrong: every node downstream of a multi-item source already loops. Wrapping it explicitly is a no-op.

The solution we're shipping

Three workflow JSONs that make the items model visible and falsifiable in under 5 minutes:

  • broken-loop.json — reproduces the "only first item works" symptom
  • fixed-loop.json — same goal, correct topology, all 5 items hit
  • three-keys-error.json — exact reproduction of the sub-workflow keys error

Plus a ROOT_CAUSE.md that explains n8n's items / fan-out semantics in <500 words with an ASCII or Mermaid diagram.

Stack

Vanilla n8n, single container. No external dependencies — the value is in the workflow JSON and the explanation.

Done when

  • All 3 workflow JSONs export cleanly via n8n's UI
  • ROOT_CAUSE explains the items model with a diagram
  • Cites the n8n staff "improve Loop Over Items" forum thread

03
Wave 1 · Day 2 Budget ~3h

Docker persistence & encryption-key drift

Two compose files · Broken vs fixed

The problem

Customer runs n8n in Docker, restarts the container, and finds their credentials are gone — or worse, present but unreadable ("decrypt failed"). Or they try to write a file from a workflow and get EACCES.

Three independent root causes converge on the same symptoms: no persistent volume, an unset N8N_ENCRYPTION_KEY that drifts every boot, and a UID mismatch between the container user and the host bind-mount owner.

The solution we're shipping

Two minimal compose files that demonstrate cause and fix side-by-side:

  • docker-compose.yml — bind-mounts to /tmp/n8n-bad with no UID hint, no encryption key, no healthcheck
  • docker-compose.fixed.yml — named volume n8n_data, N8N_ENCRYPTION_KEY from .env, user: "1000:1000", healthcheck

WORKAROUND.md includes the SQLite → Postgres migration one-liner using pgloader or n8n's CLI export.

Stack

n8n single node + a deliberately bad host bind-mount. .env.example ships N8N_ENCRYPTION_KEY=$(openssl rand -hex 32).

Done when

  • Both stacks boot on a clean machine
  • Credential-loss step reproduces 100% of the time
  • SQLite → Postgres migration is documented & tested
Build #3
08
Wave 1 · Day 3 Budget ~3h

HTTP ECONNREFUSED from inside the container

Three sub-scenarios in one stack

The problem

Customer's workflow calls http://localhost:11434 for Ollama, an internal corporate API, or an HTTPS endpoint with a self-signed cert. They see ECONNREFUSED, ETIMEDOUT, or TLS errors and assume n8n is broken.

Three near-identical symptoms with three very different fixes: localhost inside a container is the container itself; corporate networks need HTTP_PROXY/HTTPS_PROXY exported into the n8n process; and self-signed certs need a CA bundle, not a flag.

The solution we're shipping

One compose stack with n8n + a fake corporate proxy (Squid) + a fake internal-only API. Three workflows isolate each scenario:

  • 1-ollama-localhost.json → fix is http://host.docker.internal:11434
  • 2-corp-proxy.json → fix is exporting HTTP_PROXY / HTTPS_PROXY on the n8n container
  • 3-self-signed.json → fix is mounting a CA bundle (with a giant warning about NODE_TLS_REJECT_UNAUTHORIZED=0)

Each scenario fails on docker-compose.yml and succeeds on docker-compose.fixed.yml.

Stack

n8n + Squid (proxy) + a tiny internal-API container. Workflows are pre-built; reader picks which one to import.

Done when

  • All three scenarios fail deterministically on the broken stack
  • All three succeed on the fixed stack
  • Copy-pasteable env-var blocks per scenario in WORKAROUND.md

Wave 2 · Day 4 Budget ~4h

OAuth callback URL mismatch & refresh-token expiry

Real OAuth dance against a mock IdP

The problem

Customer registers an OAuth client at the provider with https://workflows.example.com/... as the callback. n8n boots with default config and emits localhost:5678/.... The OAuth dance technically completes — and then redirects nowhere useful, leaving the user staring at a blank tab.

Or: they get OAuth working, then the next workflow run after an hour fails with invalid_token. They never wired up refresh.

The solution we're shipping

Two compose stacks that run a real OAuth flow end-to-end against a containerised mock IdP — no third-party dependency:

  • Scenario 1: callback-URL mismatch reproduced and fixed via correct WEBHOOK_URL + IdP client config
  • Scenario 2: mock IdP issues a 60-second token; the next run after expiry fails; WORKAROUND.md shows how to detect and refresh

WORKAROUND.md also covers the allowedDomains Public-API gotcha and ngrok-specific guidance.

Stack

n8n + navikt/mock-oauth2-server (or Keycloak). Both stacks share the same IdP container; only the n8n config differs.

Done when

  • OAuth dance completes end-to-end on the fixed stack
  • Refresh-token expiry is reproducible by waiting 60s
  • allowedDomains + ngrok gotchas documented

05
Wave 2 · Day 5 Budget ~3h

Webhook double-trigger from provider retries

Custom flaky-provider container

The problem

Customer's webhook works fine until traffic goes up. Then the same lead shows up three times in their CRM. Their workflow is "correct" — the upstream provider is just retrying on slow responses, and without an idempotency key every retry creates a new record.

The solution we're shipping

A tiny flaky-provider Node container that simulates real provider semantics: 3 retries with exponential backoff, same payload each time, optional --slow flag to force n8n to take 10s and trigger the retry path.

Two workflows make the fix obvious by contrast:

  • non-idempotent.json → 3 retries, 3 leads created
  • idempotent.json → Set + If + Postgres dedupe table keyed on the provider's request-id header. 3 retries, 1 lead.

WORKAROUND.md covers all four common patterns: request-id dedupe, content hash, optimistic upsert, transactional outbox.

Stack

n8n + Postgres + custom flaky-provider Node container (Dockerfile + server.js in the repro folder).

Done when

  • flaky-provider reliably triggers retries
  • Non-idempotent flow double-effects 100% of the time
  • Idempotent flow correctly deduplicates

04
Wave 3 · Day 6 Budget ~5h · Hardest

Queue-mode "ghost trigger" duplicates

Main + 2 workers + Redis + Postgres

The problem

Customer scales to queue mode for performance. Their cron-triggered workflow now fires three times per tick instead of once, because the trigger registered on every worker instead of only on the main process. Race conditions are non-deterministic by definition, so the customer also can't get a consistent repro on their side — they just see duplicates and start blaming n8n.

The solution we're shipping

Two compose configurations that make this race condition 100% reproducible within ~60 seconds:

  • docker-compose.yml — main + 2 workers, workers misconfigured to also load triggers. Cron every 30s → 3 inserts per tick.
  • docker-compose.fixed.yml — same stack, workers correctly configured (no production-main on workers, active workflows disabled on workers). 1 insert per tick.

Cron + a single Postgres INSERT node is the proof — reader counts rows in the events table.

Stack

n8n main + 2 workers + Redis (queue) + Postgres (data & events table seeded by postgres-init.sql).

Done when

  • Bug reproduces 100% of the time within ~60s
  • Fix eliminates duplicates verifiably (count = 1 per tick)
  • ROOT_CAUSE cites the specific n8n version where the race was introduced
  • "Isolate workflows to dedicated workers" pattern documented

Wave 3 · Day 7 AM Budget ~3h

Memory overflow on large SQL result sets

Naive vs streamed · Same data, different fate

The problem

Customer writes SELECT * FROM big_table, pipes the result into a Loop → HTTP node, and watches their container get OOM-killed at row ~30,000. They blame n8n; really they're just loading 50k rows into a single in-memory item.

The solution we're shipping

One compose stack with mem_limit: 512m on n8n and a Postgres seeded with 50,000 synthetic rows. Two workflows demonstrate the difference:

  • naive.json — SELECT *, OOM-killed within ~30s, watch docker stats climb
  • streamed.json — chunked LIMIT 500 OFFSET ${offset} in a loop, RSS stays flat, completes successfully

WORKAROUND.md covers chunked SELECT, binary-data filesystem mode, Loop Over Items batch sizes, and EXECUTIONS_DATA_SAVE_ON_* tuning.

Stack

n8n (capped at 512MB) + Postgres seeded with 50k synthetic rows via postgres-init.sql.

Done when

  • OOM kill is reproducible & visible in docker stats
  • Streamed workflow completes; RSS stays flat
  • All four mitigation knobs documented

Wave 4 · Day 7 PM Budget ~4h

RAG vector dimensions & chunking failures

Three failure modes · Tiny eval harness

The problem

Customer wires up "AI Agent + vector store + retriever" and reports "RAG isn't working." Useless to debug without ground truth — you can't eyeball whether retrieval is good. The three real failure modes are: dimension mismatch between embedding model and collection, embeddings node silently not firing on insert, and chunk sizes too small for meaningful context.

The solution we're shipping

A compose stack with n8n + Qdrant + a small known-good corpus (~50 markdown docs about cooking) + a tiny eval container that runs precision@k against fixed Q&A pairs — turning "is RAG working?" into a green/red number in <60s.

  • 1-dim-mismatch.json — OpenAI embeddings (1536) into a Qdrant collection sized 192
  • 2-no-embedding.json — workflow looks correct but no embedding API calls fire on insert
  • 3-bad-chunks.json — chunk_size=100 destroys context; fix is 500 with overlap 50

WORKAROUND.md doubles as the seed for the "RAG production checklist" deliverable in Project 8.

Stack

n8n + Qdrant + corpus folder + tiny Node eval container (RAGAS-style precision@k against qa-pairs.json).

Done when

  • Eval harness produces a green/red score in <60s
  • All three failure modes reproduce deterministically
  • Seeds the Project 8 RAG production checklist

10
Wave 4 · Day 7 EVE Budget ~2h

Snowflake / provider-node version regression

Documentation + a generic triage script

The problem

n8n ships a minor version. Customer upgrades. The Snowflake Insert node (or Salesforce, or Gemini 3.0) now silently fails on payloads that worked yesterday. Hard to ship a literal repro because Snowflake needs paid credentials, and the bug is version-dependent — not a config error.

The solution we're shipping

This repro is intentionally documentation-heavy, but it ships a piece of tooling that pays dividends every future regression:

  • REGRESSION_LOG.md — documents at least 3 real regressions across recent versions (Snowflake, Salesforce, Gemini 3.0) with the version pair and the exact diff in behaviour
  • triage-script.sh — pulls two adjacent n8n image tags, runs the same workflow against both via n8n's CLI execute mode, diffs the outputs. Generic — works for any future node regression.
  • WORKAROUND.md — "how to roll back, and how to file a bug report n8n eng will love"

Stack

Vanilla n8n with the version pinned by env var. No paid third-party services required.

Done when

  • triage-script.sh is general-purpose — not Snowflake-specific
  • 3+ real regressions documented from forum threads
  • Bug-report template links to n8n's GitHub issue form
03  /  What "Project 1 done" looks like

The bar to clear before tagging v0.1.0-repros-complete

Each individual repro must satisfy the 11-box acceptance checklist documented in build-project1.md. The project as a whole is shipped when:

Definition of done

  1. All 10 repros have every acceptance-criteria box checked (README, broken & fixed compose, .env.example, REPRO, ROOT_CAUSE, WORKAROUND, smoke test on a clean machine, KB cross-link, source-thread citations, teardown documented).
  2. Smoke pass on a fresh clone: clone to a new directory, run all 10 repros sequentially, all boot green.
  3. Each repro is cross-linked from a KB stub — full KB articles are Project 2; just the stub link counts here.
  4. Root README.md status table shows the Repros line as ✅.
  5. Git tag v0.1.0-repros-complete exists on main.

Once those five are true, Project 1 ships, and the next build queue is the 10 KB articles in Project 2. Each KB article is ~2 hours because the source content already exists inside the matching repro folder.
