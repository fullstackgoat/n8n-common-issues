01

Webhook URL & "Connection lost" — the self-hosting onboarding wall

Frequency: very high · Severity: blocking · 100k+ aggregate views

The single biggest support pattern. Users self-host n8n, fail to set N8N_HOST / WEBHOOK_URL / N8N_PROTOCOL, then hit a chain of symptoms: webhooks stuck on localhost:5678, OAuth callbacks failing, "Connection lost" banner, websocket drops behind reverse proxies. Often compounded by Cloudflare/ngrok/nginx misconfigurations.

Root causes

  • Missing or wrong WEBHOOK_URL env var
  • Reverse proxy not forwarding Upgrade/Connection headers
  • N8N_PROXY_HOPS not set behind multiple proxies
  • HTTPS termination mismatch between proxy and n8n

Evidence

02

OAuth 2.0 credential failures (Google, Microsoft, custom)

Frequency: very high · Severity: blocks integrations · 50k+ aggregate views

Auth is the second-largest pattern, and it's a direct JD core requirement. Users hit callback-URL mismatches, refresh-token expiry on Google/Microsoft, the Public-API "allowedDomains" contradiction, ngrok loop-back failures, and "Could not connect to your MCP server" SSE auth errors. These tickets need a methodical request-trace to resolve.

Root causes

  • Callback URL points to localhost:5678 in self-hosted
  • OAuth client misconfigured (scopes, consent screen, app-mode)
  • Token refresh failing silently — provider revoked or user reset
  • allowedDomains rules conflicting via Public API
  • ngrok URL not added to OAuth client allow-list

Evidence

03

Docker self-hosting: persistence, permissions, upgrades, migration

Frequency: very high · Severity: data-loss risk · 30k+ aggregate views

Direct hit on the JD's "Docker or containerized environments" core requirement. Tickets cluster around: I lost my credentials after a restart, file is not writable, SQLite corrupted, how do I migrate to Postgres, how do I upgrade docker-compose safely, and n8n isn't reachable from another computer.

Root causes

  • Volume not mounted, or mounted to host path with wrong UID/GID
  • Encryption key lost between container restarts → credentials unreadable
  • SQLite chosen for production — corrupts under concurrency
  • Upgrade procedure run without backing up .n8n
  • Container bound to 127.0.0.1 instead of 0.0.0.0

Evidence

04

Queue mode & worker scaling regressions

Frequency: high · Severity: production-impact · IC3 differentiator

The hardest cluster — and the one IC3 hires are expected to own. Race conditions in v2.x queue mode, ghost cron triggers firing N times across N workers, "Worker failed to find data for execution" errors, UI failing to render in queue mode, Redis connection drops on Cloud Run / DigitalOcean, and Postgres-node failures inside workers.

Root causes

  • Cron trigger registered on every worker instead of only main
  • Race between Redis-published job and binary-data write to disk
  • Misconfigured EXECUTIONS_MODE=queue with main process also picking jobs
  • Redis connection limits / sentinel/cluster not supported
  • Worker concurrency too high for memory budget

Evidence

05

Webhook delivery, idempotency & the "double trigger" problem

Frequency: high · Severity: data-correctness · Direct JD core

Webhooks & triggers misfiring is the most "Slack-channel-on-fire" category. Symptoms: WhatsApp Cloud API webhook delays of minutes-to-hours, Gmail Search re-firing after success, Slack approval flows triggering twice, Google Apps Script intermittent 403s, idempotency & checkpointing gaps when a workflow partially executes.

Root causes

  • No deduplication key on webhook payloads
  • Provider retries on a 5xx because n8n didn't respond fast enough
  • Cron + manual trigger both active during testing
  • Workflow not idempotent → side effects double on retry
  • Test vs. Production webhook URL confusion

Evidence

06

AI Agent / RAG production failures

Frequency: high & rising · Severity: feature-blocking · Fastest-growing category

n8n is positioning as "the AI orchestration of your wildest imagination" — and the forum reflects it. The recurring tickets: AI Agent latency at 16s+, embeddings node silently not calling OpenAI, vector dimension mismatches, Supabase Vector Store returning empty arrays, "Received tool input did not match expected schema," and MCP/SSE connection failures.

Root causes

  • Embedding model dimension ≠ vector store column dimension
  • Document loader / splitter producing fragmented context
  • Tool schema doesn't match the LLM's output JSON
  • SSE/MCP transport blocked by reverse proxy buffering
  • Token budgets / context window exceeded silently

07

Loops, iteration & data-shape errors

Frequency: very high · Severity: workflow won't run · ~50k aggregate views

The single most teachable ticket category — these are nearly always user-error and almost always KB-able. Loop Over Items only processing the first item, "Input values have 3 keys, you must specify an input key", undefined data after a loop, and APIs that 400 only on the second iteration of a loop because the previous payload mutated state.

Root causes

  • Misunderstanding of n8n's "items" pattern (each item independent)
  • Sub-workflow returning multiple keys without an input-key picker
  • Set node leaking state between iterations
  • HTTP node sending stale headers from prior loop iteration

Evidence

08

HTTP Request & API debugging — proxies, ECONNREFUSED, custom auth

Frequency: very high · Severity: integration-blocking · Direct JD core

The Senior Support Engineer's bread & butter. Customers paste an HTTP Request node, get ECONNREFUSED, and need someone to walk through DNS → proxy → TLS → headers → body. Includes corporate proxies, local Ollama on a different host, AWS Bedrock SigV4, Bybit HMAC signatures, and self-signed certs against MS SQL Server.

Root causes

  • Container can't reach localhost (it's the container's localhost)
  • Corporate proxy not set in HTTP_PROXY/HTTPS_PROXY
  • HMAC/SigV4 string-to-sign assembled wrong
  • Self-signed cert rejected — needs NODE_TLS_REJECT_UNAUTHORIZED=0 (with caveats) or CA bundle
  • Body sent as JSON when API expects form-encoded

Evidence

09

Performance, memory & large-data crashes

Frequency: medium · Severity: production-incident · IC3 differentiator

The "is n8n down or is my workflow slow?" cluster. Workflows OOM-killed when processing 40k+ SQL rows, AI Agent extremely slow, qdrant data loader degrading on large datasets, RAM never released in queue mode, and webhook concurrency under load.

Root causes

  • Loading entire result set into one item instead of streaming/paginating
  • Binary data stored in memory instead of filesystem mode
  • Worker concurrency × memory-per-execution exceeds container limit
  • Synchronous chain of LLM calls instead of parallel

Evidence

10

Provider-specific node bugs & regressions after upgrade

Frequency: medium · Severity: integration broken on upgrade

The "it worked yesterday" cluster. A version bump breaks Snowflake Insert, an official node disappears from search results, Salesforce Case creation suddenly requires a Type Name, Gemini 3.0 tool-use returns "missing thought_signature," WhatsApp node behaves oddly with permanent API keys. Each one is a triage-and-escalate ticket.

Root causes

  • Provider API change not yet reflected in n8n's node
  • n8n release introduced a regression in a specific node
  • Required field changed upstream without notice
  • Auth flow updated by provider (e.g., new permission scopes)
